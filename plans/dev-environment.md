# dev-environment.md

Status: reference

Local development services and dev-shell environment.

## Postgres

Project-local cluster, UNIX-socket only. No system services, no port conflicts, version pinned by the flake.

### Layout

```
.pg/
├── data/      # cluster (initdb output) — gitignored
├── log/       # postgres logs           — gitignored
└── sock/      # UNIX socket             — gitignored

config/backend/db-url    # generated by pg-init, gitignored
```

### Connection settings

Set in the dev shell (`flake.nix` `shellHook`):

| Var | Value |
|---|---|
| `LUCIANA_ROOT` | `$PWD` (captured at shell entry) |
| `PGDATA` | `$LUCIANA_ROOT/.pg/data` |
| `PGHOST` | `$LUCIANA_ROOT/.pg/sock` |
| `PG_LOG` | `$LUCIANA_ROOT/.pg/log` |
| `PGDATABASE` | `luciana` |
| `PGUSER` | `luciana` |

`PGHOST` pointing at a directory makes `libpq` and `psql` connect over the socket — no password, no `pg_hba.conf` work.

### Scripts (provided by the dev shell)

| Command | Effect |
|---|---|
| `pg-init` | First-time setup. `initdb` if `$PGDATA` is empty, start, `createdb $PGDATABASE`, write `config/backend/db-url`, stop. Idempotent. |
| `pg-up` | Start the cluster (`pg_ctl start`). |
| `pg-down` | Stop the cluster (`pg_ctl stop`). |
| `psql` | Postgres `psql`, picks up `PGHOST`/`PGDATABASE`/`PGUSER` automatically. |

All three are exposed via `pkgs.writeShellScriptBin` in `flake.nix` so they live on `$PATH` inside `nix develop` / `direnv` and version with the flake.

### Backend connection

`config/backend/db-url` contains:

```
postgresql:///luciana?host=/abs/path/to/repo/.pg/sock
```

The backend reads it via `Obelisk.ExecutableConfig.Lookup.getConfig "backend/db-url"` at startup and hands it to `Database.PostgreSQL.Simple.connectPostgreSQL` (under `beam-postgres`). The file is gitignored because the absolute path is per-checkout; `pg-init` regenerates it.

### Typical dev loop

```bash
direnv allow              # enters the flake shell
pg-init                   # once per checkout
pg-up                     # at the start of a session
ob run                    # backend connects via the socket
pg-down                   # at end of session (optional)
```

## NIX_PATH (pinned to obelisk's nixpkgs)

`obelisk-command` 0.9.0.1 hard-codes a bundled Nix 2.11.0. Inside `ob run` it
performs auxiliary evaluations against `<nixpkgs>` (e.g.
`(import <nixpkgs> { system = "x86_64-linux"; }).writeText …`). Modern
nixpkgs (≥ 24.11) refuses to evaluate under Nix < 2.18, so any user whose
`NIX_PATH` resolves `nixpkgs` to a recent channel hits:

```
This version of Nixpkgs requires an implementation of Nix with the
following features: builtins.nixVersion reports at least 2.18
You are evaluating with Nix 2.11.0
```

Fix: in `flake.nix` `shellHook`, override `NIX_PATH` to obelisk's pinned
nixpkgs (which is by definition compatible with Nix 2.11):

```bash
export NIX_PATH="nixpkgs=$(nix-instantiate --eval --strict \
  -E '(import ./.obelisk/impl {}).nixpkgs.path' | tr -d '\"')"
```

Run by direnv on shell entry. Side effect: `<nixpkgs>` from inside the dev
shell now resolves to obelisk's pinned (older) nixpkgs, not the system
channel. That's fine — nothing else in this project's tooling consults
`<nixpkgs>` directly.

## Out of scope

- A second service (Redis, MinIO, etc.). If we ever add one, revisit and consider migrating to `devenv.sh` for `devenv up`-style supervision.
- CI Postgres: GitHub Actions / NixOS test will spin up its own; the local scripts aren't reused there.

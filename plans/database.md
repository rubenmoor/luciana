# database.md

Status: partial

How the backend connects to Postgres at runtime, and how schema migrations get applied. The schema itself (tables, columns, Haskell mirror) lives in `schema.md`; this file is about the runtime/migration plumbing.

---

## Existing state

### Postgres in dev (already wired)

A project-local cluster is provisioned by `flake.nix` (see `dev-environment.md` for the user-facing summary). What's actually in place today:

- **Server**: `pkgs.postgresql_16`, version pinned by the flake. No system Postgres is used.
- **Cluster location**: `.pg/{data,log,sock}` under the repo root (gitignored). UNIX-socket only — `PGHOST` is the socket directory, so no TCP port, no `pg_hba.conf` edits, no password.
- **Env vars exported by the dev shell**: `LUCIANA_ROOT`, `PGDATA=$LUCIANA_ROOT/.pg/data`, `PGHOST=$LUCIANA_ROOT/.pg/sock`, `PG_LOG=$LUCIANA_ROOT/.pg/log`, `PGDATABASE=luciana`, `PGUSER=luciana`.
- **Helper scripts** (`pkgs.writeShellScriptBin` in `flake.nix`):
  - `pg-init` — `initdb` (idempotent), start, `createdb luciana`, write `config/backend/db-url`, stop.
  - `pg-up` / `pg-down` — `pg_ctl start` / `stop`.
  - `psql` — vanilla, picks up the env vars.
- **Connection string**: `pg-init` writes `postgresql://$PGUSER@/$PGDATABASE?host=$LUCIANA_ROOT/.pg/sock` (e.g. `postgresql://luciana@/luciana?host=/abs/path/to/.pg/sock`) to `config/backend/db-url`. The path is absolute and per-checkout, so the file is gitignored and regenerated by `pg-init`. The earlier userless form (`postgresql:///luciana?...`) caused libpq to fall back to the OS user — fixed in `flake.nix` by embedding `$PGUSER`. Existing checkouts must either re-run `pg-init` or hand-edit `config/backend/db-url`.

### Backend Haskell side (already wired)

- `backend.cabal` depends on `beam-core`, `beam-migrate`, `beam-postgres`, `postgresql-simple`, `obelisk-executable-config-lookup`.
- `default.nix` pins `beam-migrate` via `callHackageDirect`.
- `Backend.Schema.{User,Session,PeriodEntry,PushSubscription,NotificationPref}` define the `*T` records; `Backend.Schema.Db` exposes `LucianaDb`, `lucianaDb`, and `checkedLucianaDb = defaultMigratableDbSettings`.

### What's missing

`Backend.hs` is still the Obelisk stub:

```haskell
backend = Backend
  { _backend_run = \serve -> serve $ const $ return ()
  , _backend_routeEncoder = fullRouteEncoder
  }
```

There is no DB connection, no migration step, and no place that consumes `config/backend/db-url`. `defaultMigratableDbSettings` also can't express the constraints/indexes called out in `schema.md` (cascade deletes, `CHECK` constraints, composite indexes, btree on `expires_at`/`user_id`).

## Plan

Goal: on backend startup, open a connection pool, bring the schema up to date, then serve. Operator gets an audit trail in production.

### 1. Wire up a connection pool

New module `Backend.Db`:

- `data DbPool = DbPool (Pool Connection)` — wrap `Data.Pool` over `postgresql-simple`'s `Connection`.
- `withDbPool :: ByteString -> (DbPool -> IO a) -> IO a` — read the URL, create a pool (size ~10), `bracket` over `destroyAllResources`.
- `runBeam :: DbPool -> Pg a -> IO a` — `withResource` + `runBeamPostgres` (or `runBeamPostgresDebug putStrLn` under a `LUCIANA_DEBUG_SQL` env flag).

The URL comes from `Obelisk.ExecutableConfig.Lookup.getConfig "backend/db-url"`. Trim trailing newline; fail loudly if absent.

### 2. Migration module

New module `Backend.Schema.Migration`:

- `initialMigration :: CheckedDatabaseSettings Postgres LucianaDb -> Migration Postgres (CheckedDatabaseSettings Postgres LucianaDb)` — start from `checkedLucianaDb` and add the bits `defaultMigratableDbSettings` won't infer:
  - `ON DELETE CASCADE` on every FK to `users` (sessions, period_entries, push_subscriptions, notification_prefs).
  - `CHECK (locale IN ('de','en'))` on `users.locale`.
  - `CHECK (mode IN ('Daily','YellowRed','RedOnly'))` on `notification_prefs.mode`.
  - `CHECK (end_date IS NULL OR end_date >= start_date)` on `period_entries`.
  - Indexes via raw SQL hooks (`runNoReturn`):
    - `CREATE INDEX period_entries_user_start_idx ON period_entries (user_id, start_date DESC)`
    - `CREATE INDEX sessions_expires_at_idx ON sessions (expires_at)`
    - `CREATE INDEX push_subscriptions_user_id_idx ON push_subscriptions (user_id)`
- `migrationSteps :: MigrationSteps Postgres () (CheckedDatabaseSettings Postgres LucianaDb)` — single step "0001_initial" wrapping `initialMigration`. Future migrations chain via `migrationStep "0002_..." (\_ -> ...)`.

Future schema changes go here as additional `migrationStep`s; never edit a past step.

### 3. Apply migrations on startup

In `Backend.hs`, replace the stub `_backend_run`:

```
_backend_run = \serve -> do
  url      <- loadDbUrl
  applyMode <- readMigrationMode  -- env var, see §4
  withDbPool url $ \pool -> do
    runMigrations pool applyMode
    serve $ \_route -> pure ()  -- handlers come in later plans
```

`runMigrations` (in `Backend.Schema.Migration`):

- Dev: call `bringUpToDateWithHooks` (or `runMigrationSilenced` for the simple case) inside `runBeamPostgres`. Idempotent: no-op when already current.
- Prod-dry-run: compute the pending diff with `verifySchema` / `collectChecks`, print the SQL to stdout, exit non-zero if any pending steps.
- Prod-apply: same but actually run it inside a single transaction. Log each step.

### 4. Operator switch

Distinguish dev vs. prod by env var, not build flag:

| `LUCIANA_MIGRATIONS` | Behavior |
|---|---|
| unset / `auto` | Apply silently. Dev default. |
| `print` | Print pending SQL, exit 0 if none, exit 1 if pending. |
| `apply` | Print + apply in a transaction. Production deploy step. |

The deploy script runs the backend once with `LUCIANA_MIGRATIONS=print` (gate), then again with `apply` (do it), then starts it normally. Aligns with the `--apply-migrations` intent in `schema.md` § Migrations without adding a CLI parser.

### 5. Verification

- `ob run` after `pg-up` should bring up an empty DB to head with no errors; second run is a no-op.
- `psql -c '\d+ users'` etc. shows the expected columns, FKs (`ON DELETE CASCADE`), and `CHECK` constraints.
- `psql -c '\di'` shows the three extra indexes.
- Add a smoke test in the backend's executable: insert a `User`, read it back, delete it, assert cascade fires for a dependent `Session`.

### 6. Out of scope (revisit later)

- Down-migrations / rollback. beam-migrate supports them but v1 will only ever go forward; restore from backup if a deploy goes wrong.
- `EXCLUDE USING gist` on `period_entries` (already deferred in `schema.md`).
- Connection pool tuning, statement timeouts, read replicas.
- Test database fixtures — the integration-test plan will own that.
